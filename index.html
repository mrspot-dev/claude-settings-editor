Standardpfade — Klick kopiert Pfad

Das Standard-Modell für Claude Code. Aliases zeigen immer auf die neueste Version — wenn Anthropic ein neues Modell released, wechselt der Alias automatisch. Leer = Opus (das stärkste Modell, Standard bei Max-Abo).

Wie viel Denkaufwand Claude in Antworten steckt. Bei "low" antwortet Claude knapp und schnell — gut für einfache Fragen. "high" aktiviert tieferes Reasoning mit mehr Kontext-Analyse — ideal für Code-Reviews, Architektur-Entscheidungen und Debugging.

Steuert den Stil der Antworten. Plugins wie "Explanatory Output Style" oder "Learning Output Style" registrieren eigene Styles, die man hier per Name aktiviert. Der Style beeinflusst Ausführlichkeit, Struktur und Tonalität — nicht die Fähigkeiten.

Die Sprache, in der Claude antwortet. Leer = Englisch.

Aktiviert den "Thinking"-Modus permanent. Claude zeigt dann seinen internen Denkprozess vor jeder Antwort — hilfreich um nachzuvollziehen, wie Entscheidungen getroffen werden. Verbraucht mehr Tokens, gibt aber maximale Transparenz.

Welchen Release-Kanal Claude Code für automatische Updates verwendet.

Priorität: deny schlägt ask schlägt allow. Was in Deny steht, wird immer blockiert — egal was Allow sagt.

Syntax: Nur Tool-Name = gesamtes Tool. Mit Pattern = nur bestimmte Aufrufe.

Bash(npm run *) — alle npm-Scripts
Read — alle Dateien lesen
Read(./.env*) — .env blockieren
Read(~/.ssh/*) — SSH-Keys blockieren

Pfad-Prefixe (Read/Edit/Write): /path = Projekt-Root, ~/path = Home-Dir, //path = Absoluter Pfad, ./path = CWD

Tipp: Viel in Allow + Gefährliches in Deny = schnelles Arbeiten ohne Risiko. Nutze Ask für Dinge, die du vor Ausführung prüfen willst (z.B. Deploys, Pushes).

Schnellstart-Vorlagen nur für Permissions. Ändert nur Allow/Ask/Deny und Default Mode — alle anderen Settings (Plugins, Hooks, etc.) bleiben unberührt.

Fallback, wenn keine Allow/Ask/Deny-Regel greift. allowPrompt = fragt immer (sicher, langsam). acceptEdits = Read/Edit/Write auto-erlaubt, Bash fragt (beste Balance). acceptAll = nichts fragt (gefährlich). plan = nur lesen, kein Schreiben.

Der Bypass-Modus erlaubt es normalerweise, alle Permissions temporär zu überspringen. Wenn deaktiviert, kann niemand diesen Modus nutzen — sinnvoll für Teams oder Shared-Configs, wo strikte Kontrolle gewünscht ist.

Tools und Patterns die Claude automatisch ausführen darf. Bash(npm run *) erlaubt alle npm-Scripts, Read ohne Pattern erlaubt das gesamte Tool. Je mehr hier steht, desto weniger Bestätigungen brauchst du. Wildcards (*) matchen beliebige Zeichen inkl. Leerzeichen.

Für Aktionen, die du vor Ausführung prüfen willst. Claude zeigt dir den Befehl und wartet auf dein OK. Ideal für Deploys, Force-Pushes und andere Dinge mit externem Effekt. Überschreibt Allow-Rules, wird aber von Deny überschrieben.

Harte Blockade — Claude kann diese Aktionen nicht ausführen, auch nicht mit Bestätigung. Deny überschreibt alles andere. Hierher gehören destruktive Befehle (rm -rf, sudo, dd), System-Manipulation (chmod, shutdown) und Zugriff auf sensible Dateien (.env, ~/.ssh).

Zusätzliche Verzeichnisse, auf die Claude zugreifen darf. Standardmäßig sieht Claude nur den aktuellen Projektordner. Hier kannst du z.B. Shared Libraries, Dokumentation oder andere Repos freigeben. Relative Pfade (../docs/) und Home-Pfade (~/shared/) funktionieren.

Hooks — Automatisierung per Event

Hooks führen automatisch Shell-Befehle oder Webhooks aus, wenn bestimmte Events eintreten. Typische Anwendungen:

  • → PostToolUse + Edit: Auto-Linting nach jeder Dateiänderung
  • → PreToolUse + Bash: Gefährliche Befehle abfangen und blockieren
  • → SessionStart: Projekt-Kontext laden oder Benachrichtigung senden

Jeder Hook braucht einen Matcher (welches Tool betroffen ist) und einen oder mehrere Actions (was ausgeführt wird). Async-Hooks laufen im Hintergrund und blockieren Claude nicht.

MCP Servers — Externe Tools per Protokoll

MCP (Model Context Protocol) erlaubt Claude, externe Tools zu nutzen — z.B. Datenbanken, Browser, APIs. Jeder Server wird als eigener Prozess (stdio) oder via HTTP/SSE-Endpoint angebunden.

stdio: Lokaler Prozess via command + args  |  http/sse: Remote-Server via URL + optional Headers/OAuth

Sandbox — Dateisystem & Netzwerk isolieren

Die Sandbox schränkt ein, auf welche Dateien und Netzwerk-Ressourcen Claude zugreifen kann. Wenn aktiviert, werden alle Bash-Befehle und Dateizugriffe in einer isolierten Umgebung ausgeführt.

Pfad-Prefixes: // = absoluter Pfad, ~/ = Home-Verzeichnis, ./ = relativ zum Projekt. Sinnvoll wenn Claude in sensiblen Umgebungen läuft oder du verhindern willst, dass z.B. ~/.ssh oder ~/.aws gelesen werden.

Alle Bash-Befehle und Dateizugriffe werden in einer Sandbox ausgeführt.

Domains, auf die Claude zugreifen darf. Wildcards möglich (z.B. *.npmjs.org).

Erlaubt Prozessen innerhalb der Sandbox, auf lokale Ports zu binden (z.B. Dev-Server).

Umgebungsvariablen — Session-Konfiguration

Variablen die in jeder Claude Code Session gesetzt werden und von allen Bash-Befehlen gelesen werden können. Bestehende System-Variablen mit $VAR_NAME referenzieren.

Nützlich um z.B. das Subagent-Modell zu steuern (CLAUDE_CODE_SUBAGENT_MODEL), Node-Umgebungen zu setzen (NODE_ENV), oder Prompt-Caching zu konfigurieren.
